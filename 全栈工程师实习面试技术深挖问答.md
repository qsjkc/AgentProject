# 全栈工程师实习面试技术深挖问答

> 适用方向：AI 硬件配套 APP、儿童 AI 产品、React Native / React、NestJS / Node、全栈工程师实习。  
> 使用方式：先背每节的“主回答”，再准备“追问展开”。面试时不要逐字背，按“结论 -> 代码事实 -> 设计取舍 -> 可改进点”回答。

## 0. 总体答题策略

### Q1. 面试官深挖技术时，回答结构应该是什么？

**建议回答：**

我会按四层回答：

1. **先给结论**：这个模块解决什么问题，核心链路是什么。
2. **再讲代码事实**：入口文件、关键函数、数据流、依赖关系。
3. **再讲设计取舍**：为什么这么设计，有什么好处和代价。
4. **最后讲改进点**：真实产品里还能怎么增强稳定性、安全性、性能或可观测性。

比如讲桌宠聊天链路，我不会只说“接了大模型”，而会讲：桌面端通过 `desktopApi.sendMessage()` 请求后端 `/chat/message`，后端经过鉴权、会话创建、用户消息保存、persona prompt / RAG context / compact prompt 拼装、LLM 调用、assistant 消息落库，最后返回 `content/session_id/sources`。

**追问展开：**

- 如果问到不熟的具体框架 API，先承认边界，再讲同类能力的迁移路径。
- 如果问“你到底做了什么”，按真实参与程度回答：开发、二次开发、运行验证、文档梳理、源码学习要区分。
- 不要把 AgentProject 包装成“我独立从零开发了全部”，除非这是真实情况。

## 1. 项目架构深挖

### Q2. 这个桌宠项目的整体架构是什么？

**建议回答：**

这个项目可以拆成四层：

- **desktop/**：Electron 桌面端，是核心用户入口，包含桌宠窗口、快捷聊天窗口、主面板窗口、托盘菜单、开机启动、窗口位置持久化和本地状态。
- **frontend/**：React + Vite Web 门户，主要负责登录注册、账户、下载、管理等页面，不是当前最核心的聊天入口。
- **backend/**：FastAPI 后端，承担 API、鉴权、用户偏好、聊天、RAG、LLM、管理员和安装包版本发布。
- **deploy/** + `docker-compose.yml`：生产部署层，包含 PostgreSQL、backend、nginx；nginx 承载前端静态资源、代理 API，并提供下载目录。

它不是一个单页 demo，而是“桌面 AI 伴随助手 + 云端账号/API + 私有知识库 + 下载发布 + 管理后台”的全链路项目。

**追问展开：**

- 用户侧主链路：桌面端交互 -> 后端 API -> DB/RAG/LLM -> 返回渲染。
- 发布链路：Electron build 生成安装包 -> 放入 `data/downloads` -> `/public/version/win-x64` 返回版本和下载地址 -> `/download/*` 提供文件。
- 部署链路：nginx 对外暴露 80，反代 `/api` 到 backend，下载文件走静态目录。

### Q3. 为什么说这个项目能体现全栈能力，而不是只有前端？

**建议回答：**

因为核心功能跨了多个边界。桌宠聊天不是只在 UI 展示一段文本，它涉及：

- 客户端：Electron 多窗口、IPC、token 存储、API base URL、快捷聊天 UI。
- 后端：FastAPI 路由、JWT 鉴权、会话和消息表、LLM 调用、RAG 召回。
- 数据：用户、偏好、会话、消息、文档元数据、向量索引。
- 部署：PostgreSQL、backend 容器、nginx、下载目录和健康检查。

面试时我会强调全栈不是“每个框架都非常深”，而是能把产品功能按端、接口、数据、AI 和部署拆清楚，并能定位问题在哪一层。

**追问展开：**

- 如果问“最熟哪层”：可以答最熟完整链路和客户端/后端联调，具体 RN/NestJS 是可迁移补齐项。
- 如果问“你怎么验证”：后端 `pytest tests/test_main.py`，桌面端 `npm.cmd run build:renderer`，前端 `npm.cmd run lint/build`。

### Q4. 一次桌宠聊天请求完整经过哪些步骤？

**建议回答：**

以非流式 `/chat/message` 为例：

1. 桌面端调用 `desktopApi.sendMessage(payload)`。
2. `request()` 读取 API base URL 和 session token，把 token 放入 `Authorization: Bearer ...`。
3. 后端 `/api/v1/chat/message` 先通过 `get_current_user` 解 JWT、查用户、检查 `status/is_active`。
4. `prepare_chat_turn()` 根据 `session_id` 找会话，没有就创建会话；保存用户消息。
5. `build_history()` 按情况加入桌宠 persona prompt、RAG context prompt、compact response prompt，再加载历史消息。
6. `stream_response_chunks(..., stream=False)` 调用 LLM service。
7. 后端保存 assistant 消息，返回 `content/session_id/knowledge_used/sources`。
8. 桌面端更新 UI。

**追问展开：**

- 当前请求中 `compact_response` 不是前端截断，而是后端追加 system prompt。
- `use_rag=true` 时，后端会按 `user_id` 过滤知识库，避免不同用户知识库串数据。
- 如果 LLM provider 不可用，chat service 会返回固定兜底文案。

## 2. Backend / FastAPI 深挖

### Q6. FastAPI 应用启动时做了什么？

**建议回答：**

启动入口是 `backend/app/main.py`。`FastAPI(..., lifespan=lifespan)` 使用异步 lifespan，在启动阶段做几件事：

- 打日志，输出应用名和版本。
- 创建 `data`、`UPLOAD_DIR`、`CHROMA_PERSIST_DIR`、`DOWNLOAD_DIR`。
- 调用 `init_db()` 初始化数据库或执行迁移。
- 调用 `seed_initial_admin()`，如果环境变量配置了初始管理员，就创建管理员账号。

应用还配置了 CORS，中间通过 `settings.API_ORIGINS` 控制允许来源；挂载了 `/download` 静态目录；并提供 `/health` 和 `/health/ready` 健康检查。

**追问展开：**

- `/health` 只说明服务进程存活。
- `/health/ready` 会检查数据库 `SELECT 1` 和上传、Chroma、下载目录是否存在，更适合作容器 healthcheck。
- `docker-compose.yml` 里 backend 的 healthcheck 就访问 `http://localhost:5000/health/ready`。

### Q7. 为什么要区分 `/health` 和 `/health/ready`？

**建议回答：**

因为服务“启动了”和“能处理业务请求”不是一回事。`/health` 用来判断进程是否活着；`/health/ready` 用来判断依赖是否准备好，比如数据库是否连通、存储目录是否存在。

真实部署中，如果只用 `/health`，可能 backend 进程活着但数据库不可用，nginx 或调度系统仍然把流量打进来，用户请求就会失败。用 readiness 检查可以减少这种问题。

**追问展开：**

- 对 Kubernetes 来说可以类比 liveness probe 和 readiness probe。
- 这个项目目前在 docker-compose 里用 healthcheck 做了类似 readiness 的检查。

### Q8. 后端配置是怎么管理的？

**建议回答：**

后端使用 `pydantic-settings` 的 `BaseSettings`，配置类在 `backend/app/core/config.py`。它会从 `.env` 读取环境变量，默认 `case_sensitive=True`，多余字段忽略。

比较关键的配置包括：

- `DATABASE_URL`
- `SECRET_KEY`、`ACCESS_TOKEN_EXPIRE_MINUTES`、`ALGORITHM`
- `ZHIPU_API_KEY`、`ZHIPU_BASE_URL`、`ZHIPU_MODEL`
- `UPLOAD_DIR`、`CHROMA_PERSIST_DIR`、`DOWNLOAD_DIR`
- `MAX_UPLOAD_SIZE`、`MAX_DOCUMENTS_PER_USER`
- `CHUNK_SIZE`、`CHUNK_OVERLAP`、`EMBEDDING_DIMENSION`
- `API_ORIGINS`

这里还做了两个比较实用的 validator：`DEBUG` 可以解析字符串，`API_ORIGINS` 支持 JSON 数组字符串和逗号分隔字符串。

**追问展开：**

- `get_settings()` 用 `lru_cache` 避免重复构造配置。
- 面试中可以讲“配置和代码分离，部署时通过 env 注入”。
- 生产环境必须替换默认 `SECRET_KEY`，否则 JWT 安全性不足。

### Q9. 后端数据库初始化和迁移策略是什么？

**建议回答：**

`backend/app/models/database.py` 创建了 SQLAlchemy async engine 和 `async_sessionmaker`。初始化逻辑里会先判断数据库方言和已有表：

- 如果是 SQLite，直接 `Base.metadata.create_all`，并补一些 legacy schema 字段。
- 如果不是 SQLite 且 `AUTO_RUN_MIGRATIONS=false`，就用 metadata bootstrap。
- 如果是关系型数据库且自动迁移开启：
  - 空库：执行 Alembic `upgrade head`。
  - 有 `alembic_version`：继续 `upgrade head`。
  - 有旧业务表但没有 Alembic 版本表：先 `create_all` 补表，再 `stamp head`。

这个设计是为了兼容本地测试、旧数据库和生产迁移。

**追问展开：**

- 好处：开发和部署更顺滑，旧库不容易直接失败。
- 风险：自动迁移在生产环境要谨慎，复杂迁移最好有人工审批和备份。
- `expire_on_commit=False` 可以让 commit 后对象属性仍可访问，减少 async 场景下的懒加载问题。

### Q10. 为什么后端选择 async SQLAlchemy？

**建议回答：**

后端是 FastAPI，天然适合异步 IO。数据库、LLM 调用、文件上传都可能是 IO 密集型。使用 SQLAlchemy async 可以避免每个请求在等待数据库时阻塞事件循环，提高并发处理能力。

不过 async 不是银弹。如果某些操作是 CPU 密集或同步阻塞，比如当前 RAG 的 `index_document()` 是同步函数，在 async route 中直接执行，处理大文件时可能阻塞事件循环。真实产品里可以把索引任务放到后台任务、队列或线程池里。

**追问展开：**

- 可以提 `asyncio.to_thread`、Celery/RQ、FastAPI BackgroundTasks。
- RAG 索引、PDF 解析、embedding 都适合异步任务化。

### Q11. API 路由是怎么组织的？

**建议回答：**

总路由在 `backend/app/api/router.py`，用 `APIRouter` 聚合：

- `auth`
- `chat`
- `rag`
- `users`
- `admin`
- `public`
- `tools`

然后在 `main.py` 里 `app.include_router(api_router, prefix=settings.API_PREFIX)`，默认前缀是 `/api/v1`。这种方式把业务模块和版本前缀分开，后续做 `/api/v2` 或替换单个模块更清晰。

**追问展开：**

- 面试可以说这是典型“模块路由聚合”。
- 和 NestJS 类比：FastAPI router 有点像 NestJS controller/module 的路由边界。

### Q12. 你如何评价当前后端分层？

**建议回答：**

当前后端有比较清楚的分层：

- `api/v1/*`：HTTP 路由和依赖注入，处理请求/响应。
- `schemas/*`：Pydantic 请求和响应模型。
- `models/*`：SQLAlchemy ORM 模型。
- `services/*`：业务逻辑，比如 chat、rag、llm。
- `core/*`：配置、安全、日志、时间等基础设施。

这对面试来说很好讲，因为它不是把所有逻辑堆在路由里。比如 `/chat/message` 路由只负责接请求、注入用户和 DB，然后把核心逻辑交给 `prepare_chat_turn()`。

**追问展开：**

- 改进点：对 RAG 索引这种耗时任务增加 job 层；对 LLM 调用增加超时、重试、熔断和指标。
- 对 NestJS 迁移：可对应 module/controller/service/dto/guard。

## 3. 鉴权与安全深挖

### Q13. 登录鉴权链路是什么？

**建议回答：**

后端使用 Bearer JWT。登录成功后返回 `access_token`，客户端保存 token；后续请求在 `Authorization` header 里带 `Bearer <token>`。

后端 `get_current_user()` 会：

1. 从 OAuth2PasswordBearer 取 token。
2. 用 `SECRET_KEY` 和 `ALGORITHM` 解码 JWT。
3. 从 payload 里读 `sub` 或 `username`。
4. 如果 `sub` 是数字，就按用户 ID 查；否则按 username 查。
5. 如果用户不存在，返回 401。
6. 如果用户 `status != active` 或 `is_active=false`，返回 403。

这说明鉴权不仅看 token 是否有效，还看账号状态。

**追问展开：**

- 401：身份无法验证。
- 403：身份有效但权限或状态不允许。
- 生产环境要保护 `SECRET_KEY`，并考虑 token 过期、刷新和退出登录后的 token 失效策略。

### Q14. 密码是怎么存储的？有什么安全点？

**建议回答：**

密码不是明文存储，而是用 `passlib` 的 `pbkdf2_sha256` 哈希。`get_password_hash(password)` 里对密码做 hash，登录时用 `verify_password()` 校验。

值得注意的是代码里 `password[:72]` 截断，这常见于 bcrypt 长度限制，不过当前 scheme 是 `pbkdf2_sha256`，不一定需要这个限制。面试中可以说：项目保留了密码长度边界处理，但生产中我会进一步统一密码策略，明确最大长度、复杂度、错误次数限制和重置流程。

**追问展开：**

- 不要自己实现哈希算法。
- hash 之外还要有验证码、限流、错误次数限制、日志审计。
- 如果问 salt：passlib 的哈希方案通常会包含 salt。

### Q15. 管理员权限怎么实现？

**建议回答：**

安全模块里有 `get_current_superuser()`，它依赖 `get_current_user()`，先完成普通鉴权，再检查 `current_user.is_superuser`。如果不是超级用户，返回 403。

这个设计的好处是权限判断集中在 dependency 里，路由层可以直接声明依赖，不用每个 admin 接口手动复制权限判断。

**追问展开：**

- 当前是简单 RBAC 的超级用户模型。
- 如果业务复杂，可以扩展 role/permission 表，或者做 resource-level permission。
- 管理员禁用用户后，`get_current_user()` 会因为用户状态阻止继续访问。

### Q16. 这个项目有哪些安全风险和改进点？

**建议回答：**

我会从四块看：

1. **JWT 安全**：生产环境必须替换默认 `SECRET_KEY`；需要考虑 refresh token、退出登录后的 token 失效、token 黑名单或版本号。
2. **上传安全**：当前限制扩展名、文件大小和文档数量，但还可以增加 MIME 检查、病毒扫描、PDF 解析隔离、文件名清洗。
3. **RAG 隔离**：当前按 `user_id` 过滤 Chroma metadata，这是必要的；还要防止删除、查询、元数据泄露等越权。
4. **Electron 安全**：已有 `contextIsolation: true`、`nodeIntegration: false`、preload 暴露受控 API，这是正确方向；还可以加强 IPC 参数校验和 CSP。

**追问展开：**

- 儿童 AI 产品还要额外考虑内容安全、隐私最小化、家长控制和日志脱敏。
- 上传文档里的个人信息也可能进入 RAG，需要隐私策略。

## 4. Chat / LLM 深挖

### Q17. 聊天服务为什么要先保存 user message 再 build history？

**建议回答：**

`prepare_chat_turn()` 先创建或获取 session，然后保存当前用户消息并 commit。随后 `build_history()` 按 `created_at.asc()` 查询该会话所有历史消息，所以当前用户消息也会包含在发给 LLM 的 messages 中。

这样做的好处是：

- 即使 LLM 调用失败，也能保留用户提问，便于排查和恢复。
- 会话标题可以用首条消息生成。
- 历史消息来源统一来自数据库，减少客户端和服务端状态不一致。

**追问展开：**

- 风险：如果 LLM 失败，只保存 user message，没有 assistant message；UI 需要处理这种不完整 turn。
- 可以增加 message 状态字段，比如 `pending/failed/completed`，提升可恢复性。

### Q18. `build_history()` 拼装了哪些 system prompt？

**建议回答：**

它会按请求参数动态拼装三类 system prompt：

1. **桌宠人格 prompt**：根据 `pet_type` 选择 cat/dog/pig，对回答风格做约束。
2. **RAG context prompt**：如果 `use_rag=true` 且检索到上下文，就加入“用知识库上下文回答”的 system prompt。
3. **compact response prompt**：如果 `compact_response=true`，加入“小桌宠气泡短回答”的 prompt，要求 2-3 行或约 80 个中文字符。

然后再把历史消息追加进去。

**追问展开：**

- prompt 顺序影响模型行为：人格、知识库、短回答都属于 system 层约束。
- 如果冲突，比如用户要求详细解释但 compact_response=true，当前 prompt 里写了“unless user explicitly asks for detailed explanation”，给了模型一个例外条件。

### Q19. `compact_response` 为什么不应该只在前端做截断？

**建议回答：**

前端截断只能减少显示长度，但可能把一句话截断成不完整语义，用户体验很差。后端在 prompt 层控制，则模型生成时就知道自己处于“小桌宠气泡”场景，会主动选择更短、更直接的表达。

这体现的是 AI 产品的关键点：展示层和生成层要一起设计。不同入口应该有不同回答策略：快捷聊天要短、主面板可以完整。

**追问展开：**

- 可以用数据评估：平均回答长度、继续追问率、打开主面板率、关闭率。
- 如果做 A/B 测试，可以比较 80 字、150 字、结构化短回答的效果。

### Q20. 普通聊天和流式聊天接口有什么差别？

**建议回答：**

普通接口 `/chat/message` 返回完整 `ChatResponse`，适合简单 UI 或不需要逐字显示的场景。

流式接口 `/chat/stream` 使用 `StreamingResponse` 和 `text/event-stream`。它的 generator 会：

1. 先 prepare chat turn。
2. 先发一个 metadata 事件，包含 `session_id`、`knowledge_used`、`sources`。
3. 逐 chunk yield LLM 内容。
4. 拼完整响应后保存 assistant message。
5. 最后发一个 `done=true` 的事件。

流式接口用户体验更好，尤其是大模型回答较慢时，可以边生成边显示。

**追问展开：**

- SSE 比 WebSocket 简单，适合服务端单向推送。
- 如果要支持用户中断、双向控制、多端协同，WebSocket 会更合适。
- 当前流式中如果客户端断开，服务端如何取消 LLM 请求可以继续优化。

### Q21. 流式接口为什么先发 metadata？

**建议回答：**

因为 UI 可能需要在内容到达前就知道这次请求属于哪个 session、是否使用知识库、有哪些 sources。尤其是新会话时，客户端需要 `session_id` 来更新状态。

先发 metadata 可以减少 UI 等待，也能让用户看到“已命中知识库”的提示。

**追问展开：**

- 可以把 metadata 和 content chunk 分成不同 event type，目前是都放在 `data:` JSON 中。
- 如果做更标准的 SSE，可以加 `event: meta`、`event: chunk`、`event: done`。

### Q22. LLM Provider 是怎么抽象的？

**建议回答：**

LLM service 里有 provider abstraction。当前 `ProviderType` 只有 `ZHIPU`，如果配置了 `ZHIPU_API_KEY`，就初始化 `ZhipuProvider`。`llm_service.chat()` 根据当前 provider 转发请求，支持 stream 和 non-stream。

如果没有配置 provider，`llm_service.chat()` 会 yield 一句“LLM provider is not configured...”而不是直接崩溃。chat service 外层还会捕获异常，返回统一兜底文案。

**追问展开：**

- 好处：未来可以扩展 OpenAI、Claude、本地模型。
- 改进：provider 级超时、重试、熔断、token 统计、成本统计和模型 fallback。

### Q23. 如果 LLM 服务超时或不可用，应该怎么处理？

**建议回答：**

当前 chat service 的兜底是捕获异常后返回固定文案。真实产品里可以更细：

- 前端显示“模型暂时不可用，可稍后重试”。
- 后端区分超时、限流、鉴权失败、模型错误。
- 对流式接口，已经输出的 chunk 要处理好，避免 UI 卡在 loading。
- 记录错误日志和 request id，方便排查。
- 对儿童产品，错误提示要友好，不能输出复杂技术错误。

**追问展开：**

- 可以加 circuit breaker 防止 provider 故障拖垮服务。
- 可以加 fallback model。
- 可以把失败消息也落库，方便用户重试。

## 5. RAG / 知识库深挖

### Q24. RAG 上传链路是什么？

**建议回答：**

`/rag/upload` 做了这些步骤：

1. 校验登录用户。
2. 创建上传目录。
3. 检查扩展名，只支持 `.txt/.md/.pdf`。
4. 检查当前用户文档数量，最多 `MAX_DOCUMENTS_PER_USER`，默认 20。
5. 读取文件内容，检查大小不超过 `MAX_UPLOAD_SIZE`，默认 10MB。
6. 用 UUID 生成存储文件名，异步写入磁盘。
7. 创建 `Document` 记录，状态为 `uploaded`。
8. 调用 `rag_service.index_document()` 做解析、chunk、embedding、写入 Chroma。
9. 成功后更新 `chunk_count` 和 `status=indexed`；失败则 `status=failed`。

**追问展开：**

- 文件元数据在 SQL 数据库，向量和 chunk 在 Chroma。
- 当前索引是同步调用，真实大文件可以做后台任务化。

### Q25. RAG 的向量是怎么生成的？

**建议回答：**

这个项目没有使用外部 embedding 模型，而是自定义 hash embedding：

- `_tokenize()` 先把文本 lower。
- 英文/数字用正则取 token。
- 中文连续片段会加入整个 segment，并加入双字 token。
- `embed_text()` 对每个 token 取 MD5，映射到固定维度向量的某个 index 上，计数加 1。
- 最后做 L2 normalize。

这更像 lightweight lexical hashing，不是深语义 embedding。优点是本地、低成本、可控；缺点是语义泛化较弱，比如同义词、语义改写、复杂问法召回效果有限。

**追问展开：**

- 加分回答：这是能跑的 baseline，不是最终最优方案。
- 提升方向：外部 embedding、rerank、query rewrite、chunk title boosting、召回评测集。

### Q26. 为什么中文 token 增强重要？

**建议回答：**

中文不像英文天然用空格分词。如果只用英文风格 token，中文句子可能无法形成有效 token，召回会很差。

项目里对中文片段加入整个 segment 和双字 token，能让“产品路线图”“用户偏好”“安装包下载”这类中文查询有基本匹配能力。虽然这不是专业中文分词，但对 hash 向量 baseline 很实用。

**追问展开：**

- 更强方案：jieba、pkuseg、BERT embedding、bge/m3e 等中文 embedding 模型。
- 儿童 AI 产品中中文表达多样，最好做真实问题集评测。

### Q27. 为什么 embedding 时要把 filename 也拼进去？

**建议回答：**

`index_document()` 里对每个 chunk 做 embedding 时使用 `f"{filename} {chunk}"`。这样文件名也会进入向量表示。

好处是用户常常按文件名或标题问问题，比如“总结一下产品路线图.md”。如果 chunk 内容里没有重复出现“产品路线图”，只用正文 embedding 可能召回弱；加入 filename 后能提升标题驱动查询的命中概率。

**追问展开：**

- 更精细做法是 metadata/title 单独加权，而不是简单拼接。
- 也可以对标题、一级标题、文件名做额外字段索引。

### Q28. RAG 如何保证用户之间的数据隔离？

**建议回答：**

写入 Chroma 时 metadata 里包含 `user_id` 和 `document_id`。查询时 `build_context()` 使用 `where={"user_id": str(user_id)}` 过滤，只检索当前用户的向量。

SQL 层也按用户过滤：文档列表、删除文档都会同时匹配 `document_id` 和 `current_user.id`。所以结构化元数据和向量检索都做了用户隔离。

**追问展开：**

- 风险：如果 metadata 写错或查询忘记 where，就可能串数据。
- 可以加测试覆盖 Bob/Carol 文档隔离；当前测试里已有类似 RAG isolation 场景。
- 企业版还需要组织/团队级权限。

### Q29. 删除文档时如何处理 SQL 和 Chroma 的一致性？

**建议回答：**

`delete_document()` 会先检查文档属于当前用户，然后调用 `rag_service.delete_document(document.id)` 删除 Chroma 中对应 document_id 的 chunk，再删除磁盘文件，最后删除 SQL 记录并 commit。

这个流程可以保证用户看不到已删除文档，向量库里也不再召回它。

**追问展开：**

- 潜在风险：如果 Chroma 删除成功但 SQL commit 失败，或者文件删除失败，会出现部分不一致。
- 改进：软删除状态、后台补偿任务、事务外资源操作幂等化、删除日志。

### Q30. RAG 的 `sources` 能不能说明模型一定引用了这些内容？

**建议回答：**

不能完全说明。`sources` 表示检索阶段命中的上下文来源，说明这些内容被作为上下文提供给模型，但不能保证模型最终回答逐字引用了每个 source。

面试中要讲清这个边界。更准确的说法是：sources 提升可追溯性，但不是严格引用证明。如果要做严谨引用，需要模型回答中显式标注引用位置，或者做答案与 source 的一致性校验。

**追问展开：**

- 改进方向：引用编号、答案句子和 source span 对齐、事实一致性检查。
- 儿童产品里 sources 可以隐藏给孩子，展示给家长或老师。

### Q31. 当前 RAG 方案的主要瓶颈是什么？

**建议回答：**

主要瓶颈有四个：

1. **embedding 能力**：hash 向量是 lexical baseline，语义理解弱。
2. **chunk 策略**：固定长度 `CHUNK_SIZE=500`、`OVERLAP=50`，没有按标题、段落、语义边界切。
3. **检索排序**：只有向量召回，没有 rerank。
4. **索引执行方式**：上传时同步索引，PDF 或大文件可能阻塞请求。

**追问展开：**

- 提升优先级：先做评测集，再换 embedding/rerank，否则不知道是否真的变好。
- 可加 metadata filter：文件类型、时间、标签、文档集合。

## 6. 数据库与数据建模深挖

### Q32. 当前核心数据表有哪些？

**建议回答：**

根据 `database.py`，核心表包括：

- `users`：账号、邮箱、密码 hash、状态、是否超级用户。
- `user_preferences`：桌宠类型、快捷聊天开关、气泡频率。
- `verification_codes`：邮箱验证码、用途、过期时间、是否已消费。
- `chat_sessions`：会话标题、用户 ID、时间。
- `chat_messages`：消息 role、content、session_id。
- `documents`：文档文件名、路径、类型、大小、chunk_count、状态。

这些表刚好对应账号、偏好、聊天、知识库四个业务域。

**追问展开：**

- 关系：User 1:N ChatSession，ChatSession 1:N ChatMessage，User 1:N Document，User 1:1 UserPreference。
- cascade delete 用于用户删除时清理会话、文档、偏好。

### Q33. 为什么消息表要单独建，而不是把历史存在 session 里？

**建议回答：**

单独的 `chat_messages` 表更适合增长型数据：

- 支持按时间排序、分页加载、删除单条消息。
- 支持 role 区分 user/assistant/system。
- 会话元数据和消息正文分离，查询会话列表时不必加载大量正文。
- 未来可以增加 token 数、状态、模型名、错误信息、引用 sources 等字段。

如果把所有消息 JSON 存在 session 里，短期简单，但长期不好查询、索引和增量更新。

**追问展开：**

- 当前 `get_sessions` 使用 `selectinload(ChatSession.messages)`，会加载消息；如果用户消息很多，列表页可以优化成只返回最后一条或消息数量。

### Q34. 如何设计 AI 硬件产品的数据表？

**建议回答：**

我会在现有用户/会话/消息基础上增加设备域：

- `devices`：设备 ID、设备类型、固件版本、最后在线时间、电量、状态。
- `device_bindings`：user_id、device_id、绑定时间、角色（孩子/家长）。
- `device_events`：连接、断开、升级、错误事件。
- `usage_events`：功能点击、会话次数、知识点、内容安全触发。
- `parent_controls`：使用时长、可用时间段、内容范围、隐私设置。

儿童产品还要遵循隐私最小化：不该采的音频、图像、位置就不要采；必须采的要说明用途并做好权限和删除机制。

**追问展开：**

- 技术重点：索引 `user_id/device_id/created_at`。
- 产品重点：家长控制、内容安全、数据透明。

### Q35. 你会如何做分页？

**建议回答：**

简单后台列表可以用 offset/limit；聊天消息、事件流这种增长型数据更适合 cursor pagination。

比如消息按 `created_at` 或 `id` 排序，请求带 `before_id` 或 `after_id`，服务端返回下一页和 `next_cursor`。这样新消息插入时，不容易出现 offset 偏移导致重复或漏数据。

**追问展开：**

- 管理后台用户列表：offset/limit 足够。
- 聊天记录：cursor 更稳。
- 事件日志：按时间 + id 双字段排序更安全。

## 7. Desktop / Electron 深挖

### Q36. Electron 主进程和渲染进程各负责什么？

**建议回答：**

主进程负责系统级能力和窗口管理，比如创建 BrowserWindow、托盘菜单、开机启动、通知、窗口位置、Electron Store、IPC handler。

渲染进程负责 UI 展示和用户交互，比如桌宠、快捷聊天、主面板。渲染进程不直接访问 Node 能力，而是通过 preload 暴露的 `desktopBridge` 调用受控 IPC。

这个项目里 `contextIsolation: true`、`nodeIntegration: false`，说明它尽量把系统能力关在主进程，通过 preload 做边界。

**追问展开：**

- preload 是安全桥，不是把所有 Node 能力暴露给页面。
- IPC 参数要校验，否则 renderer 被 XSS 后可能调用危险能力。

### Q37. 桌面端为什么要拆 pet、quick-chat、main-panel 三个窗口？

**建议回答：**

三个窗口对应三种使用场景：

- `pet`：常驻桌面、透明、无边框、置顶，负责陪伴感和入口。
- `quick-chat`：轻量聊天，跟随桌宠位置弹出，适合短问短答。
- `main-panel`：完整主面板，承载登录、聊天、知识库、设置等复杂功能。

如果都放在一个窗口里，常驻体验和复杂管理会互相干扰。拆成多窗口后，可以分别控制尺寸、展示、生命周期和交互模式。

**追问展开：**

- 难点：窗口位置同步、状态同步、生命周期、焦点、置顶、关闭隐藏而不是退出。
- quick-chat 需要根据 pet 位置计算左右侧弹出，避免超出屏幕。

### Q38. 桌宠窗口有哪些关键配置？

**建议回答：**

`createPetWindow()` 里配置了：

- `frame: false`：无边框。
- `transparent: true` 和透明背景：桌宠融入桌面。
- `resizable: false`：固定形态。
- `skipTaskbar: true`：不出现在任务栏。
- `alwaysOnTop: true`：保持可见。
- `hasShadow: false`：更像桌面宠物。
- `preload` + `contextIsolation` + `nodeIntegration: false`：安全桥接。

这些配置共同服务于“桌面宠物”体验，不是普通应用窗口。

**追问展开：**

- `setIgnoreMouseEvents` 可以控制点击穿透/交互。
- 位置要 clamp 到屏幕工作区，避免拖到屏幕外找不回来。

### Q39. 桌面端状态如何持久化？

**建议回答：**

项目使用 `electron-store`。默认存储包括：

- `petBounds`
- `petState`
- `quickBounds`
- `mainBounds`
- `autoLaunch`
- `mute`
- `apiBaseUrl`
- `sessionToken`
- `language`

窗口移动或缩放时通过 `persistBounds()` 保存位置和尺寸。这样用户重启应用后，桌宠位置、主面板尺寸、语言、token 等状态还能恢复。

**追问展开：**

- token 存在 electron-store 里，生产环境可以考虑加密存储或系统 keychain。
- 状态持久化要兼顾隐私和用户体验。

### Q40. preload 暴露了哪些能力？为什么这样设计？

**建议回答：**

`preload.cjs` 通过 `contextBridge.exposeInMainWorld('desktopBridge', {...})` 暴露受控 API，比如：

- 打开/隐藏主面板。
- 切换快捷聊天。
- 同步桌宠状态。
- 设置开机启动。
- 获取/设置 API base URL。
- 获取/设置 session token。
- 发送 renderer heartbeat。
- 获取运行状态和日志路径。

这样设计是为了让 renderer 能调用必要能力，但不直接拥有 Node 权限。它是 Electron 安全模型里的边界。

**追问展开：**

- 好的 IPC 设计应该有白名单、参数校验、错误处理和最小权限。
- 如果未来接硬件能力，也应该通过 preload 暴露有限接口，而不是让 UI 直接调用底层模块。

### Q41. quick-chat 如何跟随桌宠位置？

**建议回答：**

主进程里有 `getQuickChatAnchorPosition()` 和 `syncQuickChatPosition()`。逻辑大致是：

- 读取 petWindow bounds。
- 计算桌宠中心点，找到最近 display 的 workArea。
- 优先把 quick-chat 放在桌宠右侧，如果右侧空间不够，再考虑左侧。
- 用 `clampWindowPosition()` 保证窗口不会超出屏幕。
- 当 petWindow move 时触发 quick-chat 位置同步。

这体现桌面端交互不是简单 CSS，而是要考虑多屏幕、工作区、窗口坐标和用户拖拽。

**追问展开：**

- 如果支持多显示器，`screen.getDisplayNearestPoint()` 很关键。
- 如果窗口太大，clamp 能避免用户找不到窗口。

### Q42. 桌面端如何处理运行时调试？

**建议回答：**

主进程有 `logDesktop()`，会把 JSON line 写入用户目录下 `Detachym/logs`。renderer 可以通过 `desktop:log-debug` IPC 发调试信息，主进程也记录窗口和状态事件。还有 `renderer-heartbeat` 记录不同 view 的心跳。

这对桌面应用很重要，因为用户环境复杂，现场问题不一定能直接复现。有日志和 runtime state，才能判断窗口是否可见、token 是否存在、API 地址是否正确。

**追问展开：**

- 改进：日志级别、日志轮转、脱敏、上传诊断包。
- 儿童产品要特别注意日志不要记录敏感对话或儿童个人信息。

### Q43. Electron 安全上你会重点检查什么？

**建议回答：**

我会检查：

- `contextIsolation` 是否开启。
- `nodeIntegration` 是否关闭。
- preload 暴露能力是否最小化。
- IPC handler 是否校验参数。
- 是否有任意文件读写、任意命令执行风险。
- 是否设置 CSP，是否避免加载不可信远程内容。
- token 和隐私数据是否安全存储。

当前项目已经有 `contextIsolation: true` 和 `nodeIntegration: false`，是正确基础；后续可以加强 IPC 参数校验和 token 加密存储。

**追问展开：**

- 如果面试官问 RN，类似思路可以迁移到移动端：原生模块也要做权限边界。

## 8. Web / React / React Native 深挖

### Q44. Web 前端 API 层怎么设计？

**建议回答：**

Web 前端 `frontend/src/services/api.ts` 使用 axios create 一个实例，默认 baseURL 是 `VITE_API_BASE_URL` 或 `/api/v1`。请求拦截器从 `localStorage` 读取 token，放入 `Authorization`。响应拦截器遇到 401 会清理 token 并跳转登录页。

这说明 Web 门户使用的是典型 token-based API 封装。API baseURL 用相对路径时可以配合 Vite proxy 或 nginx 反代。

**追问展开：**

- token 放 localStorage 有 XSS 风险，可以考虑 HttpOnly cookie 或更强 CSP。
- 401 统一处理可以减少页面重复逻辑。

### Q45. 桌面端 API 层和 Web 端 API 层有什么不同？

**建议回答：**

桌面端 `desktop/src/shared/api.js` 使用 `fetch`，并通过 `desktopBridge` 获取 API base URL 和 session token。它和 Web 不同：

- Web token 存 localStorage；桌面端 token 存 electron-store，通过 IPC 获取。
- Web baseURL 多由构建环境或 nginx 决定；桌面端需要用户或安装包配置服务地址。
- 桌面端还有 `checkApiConnection()`，通过 `/public/version/win-x64` 验证服务可用。

这说明桌面客户端要面对更多运行环境差异。

**追问展开：**

- 桌面端错误提示要更清楚，比如“未配置服务地址”。
- 生产版默认 API base URL 为空，需要配置或打包注入。

### Q46. React Native 和 React Web 的可迁移能力是什么？

**建议回答：**

可迁移的是：

- 组件化思维。
- state / props / hooks。
- API 请求封装。
- 登录态和错误处理。
- 列表、表单、路由思路。
- 状态管理和业务拆分。

不可直接等同的是：

- RN 没有 DOM，布局是 Yoga/Flexbox。
- 导航、权限、App 前后台、原生模块、蓝牙/Wi-Fi、推送、本地存储都不同。
- 性能瓶颈也不同，比如长列表、图片、桥接调用、低端机适配。

**追问展开：**

- 面试时不要说“React Native 和 React 差不多”。
- 更稳妥：React 思维可迁移，但移动端工程能力要专项补齐。

### Q47. 如果让你用 React Native 做 AI 硬件 APP 首页，你会怎么设计技术状态？

**建议回答：**

我会把设备状态做成状态机，而不是多个布尔值：

```text
idle -> scanning -> connecting -> connected
                          |-> disconnected
                          |-> error
connected -> updating -> connected/error
```

每个状态对应：

- UI 文案。
- 可点击按钮。
- 下一个允许动作。
- 是否展示 loading。
- 是否允许重试。

这样可以避免出现 `isConnected=true` 但又显示“连接中”这类矛盾状态。

**追问展开：**

- 可以用 reducer 或状态机库。
- 儿童产品错误文案要面向孩子和家长，不只显示错误码。

### Q48. 如何减少 React 组件复杂度？

**建议回答：**

我会拆三类逻辑：

- 展示组件：只管 UI。
- hooks：封装请求、状态转换、副作用。
- service/API 层：封装 HTTP 和错误处理。

如果一个组件同时做请求、鉴权、状态、表单、弹窗和复杂 UI，它会很难测也难维护。可以先找重复逻辑和副作用，把它们抽到 hook 或 service。

**追问展开：**

- 不要为了抽象而抽象，只有当重复或复杂度真实存在时再抽。
- 对移动端，还要关注组件重渲染和列表性能。

## 9. NestJS / Node 迁移深挖

### Q49. 如果把 FastAPI 聊天接口迁移到 NestJS，你会怎么拆？

**建议回答：**

我会拆成：

- `ChatModule`
- `ChatController`
- `ChatService`
- `RagService`
- `LlmService`
- `AuthGuard`
- `ChatRequestDto`
- `ChatResponseDto`

Controller 负责 HTTP 层，Service 负责业务。`AuthGuard` 从 JWT 解析当前用户。`ChatService.prepareChatTurn()` 对应当前 FastAPI 的 `prepare_chat_turn()`，包含创建会话、保存用户消息、构建 prompt、调用 LLM、保存 assistant 消息。

**追问展开：**

- NestJS 里可以用 Pipe 做 DTO 校验，用 Interceptor 做日志/响应包装。
- SSE 可以用 `@Sse()` 或直接处理 response stream。

### Q50. NestJS 里的 Guard、Pipe、Interceptor 分别适合做什么？

**建议回答：**

- **Guard**：做权限判断，比如 JWT 鉴权、管理员权限。
- **Pipe**：做参数校验和转换，比如 DTO validation、类型转换。
- **Interceptor**：做横切逻辑，比如日志、响应包装、耗时统计、异常映射。

如果迁移本项目：

- JWT 校验放 Guard。
- `ChatRequest` 参数校验放 DTO + Pipe。
- 请求耗时、LLM 调用耗时、request id 可以用 Interceptor。

**追问展开：**

- FastAPI 里类似能力通过 Depends、Pydantic、middleware 实现。
- 面试要强调范式可迁移，不是只会某一个框架语法。

### Q51. Node/NestJS 里如何处理流式响应？

**建议回答：**

如果是 SSE，可以设置响应头：

```text
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive
```

然后按 `data: ${JSON.stringify(payload)}\n\n` 写入 chunk。NestJS 也可以用 `@Sse()` 返回 Observable。

关键不是语法，而是协议语义：先发 meta、持续发 chunk、最后发 done，同时处理客户端断开、错误和资源释放。

**追问展开：**

- 大模型输出要考虑 backpressure。
- 客户端断开时要取消 provider 请求，避免浪费成本。

## 10. 部署与发布深挖

### Q52. 生产部署拓扑是什么？

- `available=true`

- `available=false`

这样生产环境只需要对外暴露 nginx 的 80 端口，backend 和 postgres 不直接暴露给公网。

**追问展开：**

## 13. 算法与数据结构深挖

### Q61. 项目里哪里体现了算法和数据结构？

**建议回答：**

不是只有刷题才叫算法。这个项目里算法体现在：

- RAG 文本 chunk：滑动窗口，`CHUNK_SIZE` 和 `CHUNK_OVERLAP`。
- hash embedding：token -> md5 -> vector index -> normalize。
- 向量检索：Chroma 使用 cosine space。
- 消息历史：按时间排序。
- 窗口定位：根据屏幕 workArea 做 clamp。
- 状态机：设备连接、窗口显示、聊天 loading 都适合状态建模。

**追问展开：**

- 可以准备两数之和、有效括号、二分、LRU。
- 回答时要把基础结构和业务场景连起来。

### Q62. 解释一下当前 hash embedding 的复杂度。

**建议回答：**

假设文本 token 数是 `n`，embedding 维度是 `d`。当前算法对每个 token 做一次 hash 并更新向量，时间复杂度大约是 `O(n)`，最后 normalize 需要遍历向量，`O(d)`，所以总计 `O(n + d)`。

空间复杂度是 `O(d)`，因为只维护固定长度向量。这个方法成本低、稳定，但语义能力弱。

**追问展开：**

- d 默认是 `EMBEDDING_DIMENSION=128`。
- token 冲突不可避免，因为多个 token 可能 hash 到同一维。

### Q63. RAG chunk overlap 有什么意义？

**建议回答：**

固定长度切 chunk 容易把一句话或一个概念切断。overlap 可以让相邻 chunk 共享一部分文本，降低关键信息被切断的概率。

比如 `CHUNK_SIZE=500`、`CHUNK_OVERLAP=50`，每个新 chunk 会回退 50 个字符开始。这样召回时更可能拿到完整上下文。

**追问展开：**

- overlap 太小会断语义，太大会增加索引体积和重复召回。
- 更好方案是按段落、标题或语义切分。

## 16. 面试前高优先级背诵清单

### 必背 1：聊天链路

桌面端 `desktopApi.sendMessage` -> 带 Bearer token 请求 `/chat/message` -> FastAPI 鉴权 -> 创建/获取 session -> 保存 user message -> 拼 persona/RAG/compact prompt -> 调 LLM -> 保存 assistant message -> 返回 content/session_id/sources。

### 必背 2：RAG 链路

上传校验扩展名/大小/数量 -> UUID 写文件 -> SQL Document 记录 -> 解析 txt/md/pdf -> 固定长度 chunk + overlap -> 中文 token 增强 -> hash embedding -> Chroma add，metadata 带 user_id/document_id/filename -> 查询时按 user_id filter -> 返回 context 和 sources。

### 必背 3：Electron 链路

主进程创建 pet/quick-chat/main-panel 三窗口 -> electron-store 持久化 bounds/token/petState/language/apiBaseUrl -> preload 暴露 desktopBridge -> renderer 通过 IPC 调主进程能力 -> API 层通过 bridge 取 token 和 base URL 请求后端。

### 必背 4：安全边界

JWT 鉴权 + 用户状态检查；RAG 按 user_id 隔离；上传限制扩展名/大小/数量；Electron 开启 contextIsolation、关闭 nodeIntegration、通过 preload 暴露白名单 API；生产环境要替换 SECRET_KEY、加强 token 存储、IPC 参数校验和日志脱敏。

### 必背 5：承认短板的说法

我目前不是 RN/NestJS 生产经验最强的人，但我已经有 React/Electron 客户端、FastAPI 后端、API 鉴权、RAG、LLM、部署和测试的完整链路基础。React 思维、接口设计、状态管理、分层服务、联调排错可以迁移；移动端原生能力、NestJS 工程范式我会通过小 demo 和项目实战快速补齐。
