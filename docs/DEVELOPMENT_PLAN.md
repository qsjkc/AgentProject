# Detachym 开发方案

## 1. 项目现状

当前项目已经具备三层基础能力：

- Web 门户：注册、登录、账户中心、下载页、管理员后台
- 后端服务：用户体系、RAG 文档上传、聊天接口、管理员管理
- 桌面端：桌宠、快捷聊天、主面板、语言切换、偏好同步

从形态上看，这不是单纯的 Web 聊天机器人项目，而是一个“桌面常驻 AI 助手 + Web 门户 + 云端能力”的一体化产品。

## 2. 产品定位

### 推荐定位

优先把 Detachym 做成：

`桌面常驻的 AI 陪伴助手`

而不是直接做成：

`通用型 AI 平台`

### 原因

同类成熟产品已经在通用平台方向建立优势：

- `Dify`：工作流、知识库、应用编排能力成熟
- `Open WebUI`：开源聊天界面、RAG、模型接入能力完整
- `AnythingLLM`：个人/团队知识库与文档问答方向成熟
- `Cherry Studio` / `Jan`：桌面 AI 客户端、多模型与知识库能力完善

Detachym 当前最有辨识度的部分，不是“又一个聊天框”，而是：

- 桌宠常驻
- 轻量快捷交互
- 人格化陪伴感
- 桌面效率工具化

所以战略上应当避开“平台大而全”，集中打磨“桌面端高频使用场景”。

## 3. 核心发展方向

### 方向一：桌面端优先

桌面端应当成为整个产品的主入口，而不是 Web 页的附属客户端。

重点功能：

- 全局快捷键唤起快捷聊天
- 截图提问 / 选区提问
- 当前窗口上下文感知
- 剪贴板识别与一键解释
- 语音输入与语音播报
- 自动更新与版本提示

目标是把桌面端做成“随手可用”的高频工具，而不是“安装后偶尔打开”的展示型桌宠。

### 方向二：桌宠人格系统

当前桌宠已经有基础形象和台词，但还停留在轻表现层。

下一步要把人格做深：

- 每种桌宠独立人格设定
- 不同人格影响回复语气、建议方式、提醒风格
- 基于历史互动形成长期记忆
- 提供工作模式、学习模式、陪伴模式切换
- 气泡风格、台词、快捷聊天提示统一联动

目标是让“桌宠”不只是 UI 装饰，而是真正影响交互体验。

### 方向三：个人效率工具

Detachym 很适合从“陪伴”延展到“轻效率”。

建议优先引入：

- 番茄钟
- 喝水提醒 / 休息提醒
- 今日待办概览
- 晨间简报 / 晚间总结
- 轻量日程同步
- 久坐提醒、工作节奏建议

这类功能和桌宠形态天然匹配，容易形成日活和习惯依赖。

### 方向四：知识库能力增强

知识库现在更像“能上传文档”，还不够强。

建议继续补：

- URL / Sitemap 导入
- 引用来源展示
- 命中文段预览
- 分块和检索调试
- 混合检索与 rerank
- 文档标签、文件夹、分类管理
- 异步索引与任务状态追踪

目标不是追求大而全，而是让用户明确感知：

`这个 AI 回答是有依据的`

### 方向五：安全与信任机制

只要产品往“陪伴助手”方向走，就必须提前做安全底座。

建议纳入：

- 明示 AI 身份，不伪装真人
- 敏感话题和高风险内容保护
- 长时聊天提醒
- 未成年人限制策略
- 用户数据和聊天记录的隐私说明
- 管理端审计能力

这部分越晚补，后续成本越高。

## 4. 推荐产品路线

### 4.1 主线建议

优先走：

`C 端个人桌面助手`

而不是一开始就走：

`团队知识库 / 企业平台`

### 理由

- 当前桌面宠物和陪伴式交互已经具备雏形
- 个人用户更容易接受桌宠 + 快捷聊天 + 提醒的组合
- C 端可以更快验证“是否会被日常使用”
- 团队化能力需要更完整的权限、审计、协作、部署能力，成本更高

### 后续扩展

等 C 端打磨稳定后，再考虑 B 端扩展：

- 团队知识库
- 多角色权限
- 白标化
- 企业部署
- 使用审计和配额管理

## 5. 分阶段开发计划

## 第一阶段：1 个月内

目标：把桌面端打磨成真正可用的 MVP。

重点任务：

- 完成桌宠交互稳定化
- 增加全局快捷键
- 快捷聊天支持截图提问
- 增加聊天来源引用展示
- 主面板偏好项进一步完整化
- 修复文案乱码和文档编码问题
- 增加桌面端异常上报与日志机制
- 完成安装包自动更新能力设计

验收标准：

- 用户能高频使用桌面端完成快捷问答
- 桌宠交互稳定，不再出现拖拽、定位、窗口异常
- 主要页面与桌面端文案统一、中文体验完整

## 第二阶段：1 到 3 个月

目标：形成差异化体验。

重点任务：

- 完成人格系统 1.0
- 增加长期记忆
- 增加番茄钟、提醒、日程摘要
- 优化知识库检索效果
- 增加 URL 导入、引用溯源、分块预览
- 增加聊天会话标签和搜索
- 完成自动更新、版本提示、崩溃恢复

验收标准：

- 用户不再只把它当聊天工具，而是日常桌面助手
- 不同桌宠确实带来不同交互感受
- 知识库回答能给出更高可信度

## 第三阶段：3 到 6 个月

目标：验证商业化方向。

两条路线二选一：

### 路线 A：C 端增强

- 皮肤系统
- 桌宠人格市场
- 会员功能
- 多端同步
- 云记忆与个人工作空间

### 路线 B：B 端扩展

- 团队知识库
- 组织与成员管理
- 角色权限
- 白标部署
- 审计日志与配额

建议优先路线 A，等用户价值跑通后再考虑路线 B。

## 6. 工程优化方向

### 桌面端

- 加 E2E 测试，覆盖拖拽、双击、快捷聊天、窗口定位
- 引入自动更新
- 优化异常日志收集
- 补充快捷键、截图、系统通知能力

### 后端

- 文档上传和索引改为异步任务
- 建立 RAG 效果评测集
- 增加更细粒度的错误码和业务提示
- 清理 `datetime.utcnow()` 弃用问题

### 前端

- 保持中文优先体验
- 统一文案和编码格式
- 增加错误边界和空态引导
- 下载、注册、账户中心继续强化产品感

### 运维

- 增加基础监控
- 增加日志归档
- 优化部署文档
- 规划版本发布流程和回滚策略

## 7. 近期优先级建议

如果资源有限，建议按这个顺序推进：

1. 桌面端交互稳定性
2. 快捷聊天高频入口能力
3. 桌宠人格系统
4. 知识库可信度增强
5. 轻效率工具
6. 自动更新和版本管理
7. 商业化探索

## 8. 成功标准

Detachym 是否走对方向，不看“功能数量”，主要看这几个指标：

- 用户是否愿意长期把桌宠挂在桌面上
- 快捷聊天是否成为高频入口
- 用户是否能感知不同桌宠人格差异
- 知识库回答是否可信、可追溯
- 用户是否会把它当作“陪伴 + 效率工具”，而不是一次性试玩项目

## 9. 结论

Detachym 最合理的中期方向不是和通用 AI 平台正面竞争，而是：

`围绕桌面端，做一个有人格、有陪伴感、能提高效率的 AI 助手`

这条路线更符合你当前项目基础，也更容易形成产品辨识度。
